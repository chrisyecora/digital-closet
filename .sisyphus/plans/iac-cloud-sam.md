# Plan: AWS SAM Cloud Infrastructure

## Objective

Establish a robust, repeatable cloud infrastructure using AWS SAM to deploy the "Core Three" services (S3, SQS, RDS with pgvector) and automate deployments via GitHub Actions.

## Context

The `MASTER_PLAN.md` calls for real AWS infrastructure. We have already established local mocks (MinIO, ElasticMQ) in a separate phase to unblock development. This plan focuses purely on cloud deployment parity and CI/CD.

## Guardrails & Assumptions

- **Tooling**: We will use **AWS SAM** (CloudFormation).
- **Scope**: Focus on **S3, SQS, RDS**. ECS/Fargate is deferred.
- **Networking/VPC**: SAM lacks a "one-line VPC" abstraction. The template must explicitly define the VPC, Subnets, IGW, and RouteTables required for RDS within CloudFormation.
- **RDS & pgvector**: Ensure an `AWS::RDS::DBParameterGroup` is configured with `shared_preload_libraries` enabling pgvector. Add `DeletionPolicy: Retain` for S3 and RDS in production.
- **Secret Management**: Use `AWS::SecretsManager::Secret` for RDS credentials.
- **Environment Parity**: Use `samconfig.toml` to manage environment-specific parameters (`dev` vs. `prod`).
- **CI/CD Security**: The GitHub Actions workflow MUST use **OIDC** (OpenID Connect) for AWS authentication. Handle the IAM OIDC singleton carefully. No long-lived access keys.

---

## Implementation Steps

### Phase 1: Infrastructure as Code (AWS SAM Template)

- [ ] Create the directory structure: `infrastructure/sam/`.
- [ ] Write `infrastructure/sam/template.yaml` defining the VPC, Subnets (at least two across AZs), Route Tables, and Internet Gateway necessary for RDS.
- [ ] Add the S3 Bucket resource to `template.yaml` (lifecycle rules, CORS for mobile uploads, `DeletionPolicy: Retain`).
- [ ] Add the SQS Queue and Dead-Letter Queue to `template.yaml`.
- [ ] Add the RDS Postgres instance with pgvector support via `AWS::RDS::DBParameterGroup` and credentials via `AWS::SecretsManager::Secret` (`DeletionPolicy: Retain`).
- [ ] Add the `AWS::IAM::Role` and `AWS::IAM::OIDCProvider` for GitHub Actions.
- [ ] Create `infrastructure/sam/samconfig.toml` for managing environment-specific overrides (`dev`, `prod`) and SAM artifact bucket resolution.

### Phase 2: CI/CD Pipeline Configuration

- [ ] Create `.github/workflows/deploy-infra.yml`.
- [ ] Configure `aws-actions/configure-aws-credentials` to use the OIDC role defined in Phase 1.
- [ ] Implement the `on: pull_request` trigger to run `sam validate` and potentially a dry-run or linting check on the CloudFormation template using `cfn-lint`.
- [ ] Implement the `on: push` (to `main`) trigger to run `sam build` and `sam deploy --no-confirm-changeset --no-fail-on-empty-changeset`.
- [ ] Verify the SAM artifacts bucket is handled correctly via `--resolve-s3` or explicit referencing within `samconfig.toml`.

---

## Final Verification Wave

- [ ] **IaC Validation**: Run `sam validate --lint` against `infrastructure/sam/template.yaml` locally to ensure the template is structurally sound.
- [ ] **Dry Run**: Run `sam deploy --dry-run` to verify the changeset generated by CloudFormation accurately reflects the intended Core Three resources.
